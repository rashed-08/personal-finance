# Backup History Table Specification

## Purpose

The `backup_history` table records every backup and restore operation performed by the application.

This table stores backup metadata only.

It does not contain backup data.

Actual backup files are stored in local storage or Google Drive.

# Business Rules

- Every backup creates one history record.
- Every restore creates one history record.
- Failed operations are preserved.
- History is append-only.
- History records are never edited.
- History records are never deleted automatically.

# Design Decisions

- Metadata only.
- Backup files remain outside the database.
- History is immutable.
- Failures are preserved.
- Restore history is audited.
- One record per operation.

# Final Statement

The `backup_history` table provides an immutable audit trail for all backup and restore operations.

By separating backup metadata from backup files, the application maintains a clean architecture while ensuring traceability, reliability, and future extensibility.

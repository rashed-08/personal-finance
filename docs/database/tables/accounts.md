# Accounts Table Specification

## Purpose

The `accounts` table represents all financial accounts where money can be stored, transferred, received, or spent.

An account is the origin or destination of financial transactions.

Every financial transaction references one primary account.

The Accounts module owns this table.

# Business Rules

- Account names should be meaningful.
- Inactive accounts cannot receive new transactions.
- Historical transactions remain valid even after deactivation.
- Account balances are derived from transactions.
- Accounts never store running balances.

---

# Balance Calculation

# Design Decisions

- UUID is used as the primary key.
- Running balances are not stored.
- Accounts are immutable identifiers.
- Deactivation is preferred over deletion.
- Currency support is future-proofed.
- Business rules are enforced in the application layer.

---

# Final Statement

The `accounts` table provides the foundation for all financial movements within the application.

It intentionally stores only account metadata while delegating balance calculation to the transaction history, ensuring consistency, auditability, and future compatibility with ledger-based accounting.

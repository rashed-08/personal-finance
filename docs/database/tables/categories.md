# Categories Table Specification

## Purpose

The `categories` table defines how financial transactions are classified.

Categories organize income and expense transactions, enabling meaningful reporting, budgeting, analytics, and filtering.

Categories contain no financial values.

All financial calculations are derived from the `transactions` table.

The Categories module owns this table.

# Business Rules

- Every transaction belongs to exactly one category (except certain system-generated transactions if explicitly allowed).
- Categories cannot own transactions.
- Categories do not store totals.
- Deactivating a category does not affect historical transactions.
- Reports are generated from transactions, not category summaries.
- Parent categories are organizational only.

# Design Decisions

- Categories contain no financial totals.
- Category hierarchy is optional.
- Categories classify transactions only.
- Transactions remain the source of truth.
- Soft deactivation is preferred over deletion.
- Reports are always calculated from transaction history.

---

# Final Statement

The `categories` table provides the classification layer of the financial domain.

It enables meaningful reporting, filtering, and analytics while remaining independent of financial calculations.

Keeping categories lightweight ensures that business reporting remains accurate, maintainable, and extensible as the application evolves.
